# LP作成ルール（Figma非依存・事例集・サブページ向け）

メインLP（`/`）はFigmaで精密に設計されているが、**業界事例ページ（`/industries/kaigo/` 等）やその他サブLPにはデザインがない**。  
このドキュメントは、**大枠の統一感**と**メインドメイン配下の事例集としての一般常識**を定める。

---

## 1. 対象と目的

- **対象**: 業界別事例LP、キャンペーンLP、その他メインLP以外のランディングページ
- **目的**:  
  - サイト全体の見た目・体験の一貫性を保つ  
  - 恣意的な余白・フォントを減らし、保守しやすくする  
  - 事例集としてのSEO・導線・信頼性を満たす  

---

## 2. レイアウト・コンテナ

### 2.1 横方向（コンテナ）

| 対象 | モバイル | PC（768px以上） |
|------|----------|-----------------|
| 左右パディング | **16px** | **120px** |
| コンテンツ最大幅 | 制限しない（必要時のみ max-width 使用） | **1440px** |

- **原則**: 横余白は `<Container>` コンポーネントに任せる。  
  - `components/Container.tsx` が 16px / 120px を付与する。  
- セクション内で独自に `paddingLeft` / `paddingRight` を付ける場合は、**16px / 120px のみ**を使う。  
  - 例: `paddingLeft: '16px', paddingRight: '16px'`（モバイル）、`paddingLeft: '120px', paddingRight: '120px'`（PC）
- **禁止**: 20px, 24px, 32px, 40px などの「その場限り」の横パディング（コンテナとして）。

### 2.2 ブレークポイント

- **1つだけ使用**: **768px**（Tailwind の `md`）
- モバイル: `&lt; 768px`  
- PC: `≥ 768px`  
- **禁止**: 768 以外のブレークポイントを新設しない（例: 640px, 1024px）。デザイン上どうしても必要なら、例外としてコメントで理由を残す。

### 2.3 モバイル / PC の書き方

- セクションごとに見せ方を変える場合:  
  - モバイルのみ: `className="md:hidden"`  
  - PCのみ: `className="hidden md:block"` または `className="hidden md:flex"`  
- 共通マークアップで余白だけ変える場合: Tailwind の `px-4 md:px-[120px]` など、**ルールで定めた値**に揃える（後述の余白スケール参照）。
- メインLPと同様、**モバイル用ブロック**と**PC用ブロック**を分けて書くパターンでよい。

### 2.4 セクション骨組みの標準（メインドメイン配下ページ）

各セクションは次の骨組みで揃える。横方向の余白は **Container に一本化**し、セクション本体には縦余白だけ持たせる。

- **パターン**:
  ```tsx
  <section style={{ paddingTop: '60px', paddingBottom: '60px' }}>
    <Container>
      {/* 中身。横パディングは Container が 16px / 120px を付与 */}
    </Container>
  </section>
  ```
- **縦**: セクションの `paddingTop` / `paddingBottom` は **48, 60, 80 のみ**（余白スケール §3 に従う）。
- **横**: セクションや外側の div に `paddingLeft` / `paddingRight` を**付けない**。中身は必ず `<Container>` でラップし、横余白はここだけから付与する。
- **フッター**: 縦は **48px**。同じく `<Container>` でラップし、横は Container のみ。
- モバイルとPCで見せ方を分ける場合は、セクションを `md:hidden` / `hidden md:block` で分け、**それぞれのブロック内**で上記「section + Container」を守る。

この骨組みを押さえた上で、色・CTA・文言はページごとに独自に作成する。

## 3. 余白スケール（padding / margin / gap）

**恣意的な数値は使わない。以下のスケールに統一する。**

### 3.1 使用してよい値（px）

| 用途 | 使用可の値 |
|------|------------|
| セクション縦パディング（上下） | **48**, **60**, **80** |
| セクション間・ブロック間のマージン | **24**, **40**, **60**, **80** |
| 要素間の gap | **8**, **12**, **16**, **20**, **24**, **40** |
| コンテナ横（前述） | **16**（モバイル）, **120**（PC） |
| フッター縦 | **48** |
| キャプション・タグ等の小さい余白 | **4**, **8** |

### 3.2 セクション縦パディングの標準

- **モバイル**: `paddingTop: '60px', paddingBottom: '60px'` を標準とする。  
  - CTA など強調セクションは `80px` 可。
- **PC**: `paddingTop: '60px', paddingBottom: '80px'` または `80px` / `100px` の組み合わせ。  
  - メインLPでは 60/80 や 80/100 が使われている。

### 3.3 禁止

- 上記一覧にない「なんとなく」の値（例: 22px, 36px, 50px, 70px）は使わない。  
- やむを得ず使う場合は、このMDに「例外」として追記する。

---

## 4. コンテンツ幅（max-width）

- **モバイル**: 中央寄せで読みやすくする場合、`maxWidth: '343px'` を使用してよい（メインLPの CTA・Footer 等と同一）。  
- **PC**: コンテナ全体は `max-width: 1440px` で揃える。  
- それ以外の **独自の max-width（例: 500px, 800px）は原則禁止**。必要な場合はルールに追加してから使用する。

---

## 5. 画像

- **基本方針**: 画像は**表示領域（枠・アスペクト比・サイズ）だけ先に決めておき、実画像の配置は後から自分で行う**。
- LP作成時は次のようにする。
  - 画像を置くエリアを、余白スケールに合わせた幅・高さまたはアスペクト比で確保する。
  - プレースホルダー（薄い背景色＋「画像予定」などのラベル）や `next/image` の `width` / `height` だけ指定した空き領域でよい。
  - 実際の画像ファイル（`public/images/` 等）の配置・差し替えは、後から行う前提でよい。
- こうしておくことで、レイアウトと余白の統一を崩さずに、あとからデザインや素材を入れ替えやすい。

---

## 6. タイポグラフィ

### 6.1 フォント

- **日本語**: `fontFamily: '"Noto Sans JP"'` または Tailwind の `font-sans`（globals で Noto Sans JP を当てている想定）
- **数値・英字アクセント**: Inter 等、メインLPで使っている変数（`var(--font-inter)`）があればそれに合わせる。

### 6.2 使用してよいフォントサイズ（px）

- **見出し**: 24, 28, 36  
- **サブ見出し・リード**: 16, 18, 20  
- **本文**: 14, 16  
- **キャプション・補足**: 10, 12  
- **特別（Hero 等）**: メインLPで使っている 50, 70, 80 等は「トップ級」の見出しに限り使用可。

`app/globals.css` の `.text-24`, `.text-28` 等のユーティリティがある場合は、**クラスで指定する**と統一しやすい。

### 6.3 太さ

- **見出し**: `fontWeight: 700`  
- **本文・リード**: `fontWeight: 400` または `500`  
- **ラベル・キャプション**: `700` で強調可。  
- **禁止**: 800, 900 など未定義のウェイトを安易に使わない。

---

## 7. 色

- **指定しない**。色の統一感は求めない。各LP・業界ごとに自由に決めてよい。

---

## 8. 事例集LPとしての一般常識

メインドメイン配下で「事例紹介」「業界別」を扱う場合、以下を満たす。

### 8.1 共通UI

- **ヘッダー・フッター**: メインLPと同じ `Header` / `Footer` を使う。事例ページだけ別デザインにしない。
- **CTA**: 「資料請求」「無料トライアル」など、メインLPと同じ導線を用意する。ボタンは既存の `CTAButton` やメインLPのスタイルに合わせる。

### 8.2 導線・階層

- **トップへのリンク**: ロゴクリックで `/` に戻れるようにする（通常は Header で実現）。
- **事例一覧へのリンク**: 「事例紹介」「業界別のご紹介」から `/industries/` に戻れるようにする。  
  -  breadcrumb（`ホーム > 事例紹介 > 介護`）があればさらに良い。
- **1ページ1テーマ**: 1業界1ページを基本とし、**h1 は1つ**。h2 → h3 の階層を論理的にする。

### 8.3 SEO・メタ

- **title**: 「Omakase.ai 事例｜〇〇業界のAI音声・チャット接客」など、業界名＋サービス名を含める。
- **description**: そのページの内容を1〜2文で要約し、キーワードを自然に含める。
- **canonical**: そのページの正規URLを指定する（同一ドメイン・サブパスで運用する前提）。
- **OGP**: 必要に応じて `og:title`, `og:description`, `og:image` を設定する。

### 8.4 コンテンツ構成の目安

- **Hero**: 業界名＋「こんな課題に」「Omakase.aiでできること」を短く伝える。
- **課題・活用シーン**: その業界ならではのユースケースを2〜4つ。
- **CTA**: 資料請求 or 無料トライアル。ページ下部に最低1回は配置。
- **メインLPとの差別化**: 業界向けの文言・イメージにし、メインLPのコピーをそのまま長く貼らない。

---

## 9. コンポーネントの使い回し

- **Container**: 横パディング用に必ず検討する。  
- **CTAButton**: プライマリCTAに使用。  
- **Header / Footer**: そのまま使用。  
- 新規コンポーネントを作る場合: 余白・フォントは**このルールのスケール**に合わせる。色は指定しない。

---

## 10. チェックリスト（LP公開前）

- [ ] 横パディングは 16px / 120px（Container または同値）のみ
- [ ] 縦余白・gap は「余白スケール」の値のみ使用
- [ ] ブレークポイントは 768px（md）のみ
- [ ] フォントは Noto Sans JP、サイズは指定リスト内
- [ ] 同一の Header / Footer を使用
- [ ] ページに CTA（資料請求 or 無料トライアル）が含まれる
- [ ] 事例一覧（/industries/）またはトップへの導線がある
- [ ] title / description / h1 が設定され、canonical が妥当
- [ ] 画像は領域のみ先行定義でよい（実画像は後から自分で配置する前提でよい）

---

## 11. 例外と更新

- **例外**: 上記ルールを破る値・コンポーネントを使う場合は、このMDの「例外」セクションに理由とともに記載する。
- **更新**: 新しいサブLPを増やす過程で「よく使う値」ができたら、余白スケールや色にマージする。  
- メインLP（Figma由来）の既存実装は**このルールの対象外**。変更時はデザインに合わせる。

---

このルールに沿うことで、Figmaがなくても**大枠の統一感**を保ちつつ、メインドメイン配下の事例集として無理のないLPを増やしていけます。
